• After reading hibacaml_agi26.pdf and re-checking the latest CIFAR10 run, the main conclusion is sharper now:

  The current CIFAR10 experiments are not failing because the causal machinery is dead anymore. They are failing because the system is learning a good audit signal, but that signal is not changing the support
  set enough to reduce interference.

  Why that follows from the paper and the results:

  - HiBaCaML says continual learning works when each task mostly updates a small task-specific support and cross-task interference stays small.
  - The paper’s Theorem 2 is local: exact one-swap teacher corrections are guaranteed not to worsen the audited local objective.
  - The paper’s theory does not say “learn a predictor and hope.” It says the top-level controller should be corrected by local counterfactual teachers.
  - In our latest full run results/split_cifar10_columnar/split_cifar10_columnar_seed42_20260421_055724, the predictor is finally informative by task 2:
      - mix_gate: [0.0, 0.0, 0.2690, 0.2890, 0.3077]

      - task 4: shared + 24,25,26,27,34
  - Final retention is unchanged:
      - average test accuracy 0.7465
      - average forgetting 0.1731

  So in HiBaCaML terms, we have improved the teacher signal, but we are not yet using local teachers as actual teachers. The controller remains too close to a fixed sparse partitioning scheme.

  Best Proposals

  1. Make exact one-swap teaching authoritative at task boundaries.
     Use the audited one-swap objective to actually replace the initial support when an audited neighbor is better, instead of mostly using audits as supervision for a predictor. This is the most paper-aligned
     next move.
  2. Log “initial support vs teacher-improved support vs final support” for every task.
     Right now we cannot cleanly see whether the learned causal path or audit path changed anything. The paper explicitly argues for local teacher corrections and reconstructible routing decisions.
  3. Add an explicit support-change counter and overlap delta metric.
     For each task, save:

  - number of audited improving swaps
  - best one-swap gain
  - whether a swap was applied
  - overlap between initial and final support
    If those are near zero, the selector is not participating in the way the theory expects.

  4. Move from “predictive guidance only” to a hybrid controller.
     Use:

  - exact one-swap teacher at task start
  - learned causal scorer only to shortlist candidate swaps during training
    That matches the paper’s conservative logic much better than relying on learned guidance to directly move supports early.

  5. Reduce always-on shared pressure as a serious CIFAR10 ablation.
     The paper’s adequacy story is about small task-specific supports and low interference. Our supports always include the same 4 shared columns and often the same reserve column. That may be the real
     forgetting bottleneck. Test shared_columns 4 -> 3 before broader architecture changes.
  6. Run a true “teacher-only CIFAR10” ablation.
     Disable learned causal guidance entirely and do:

  - exact boundary support search if feasible, or
  - exhaustive one-swap teacher improvement at each boundary
    This will answer the core paper question: is the architecture adequate and the controller suboptimal, or is the architecture itself still too interfering?

  7. Add a joint-training upper bound and a monolithic-stem baseline.
     The paper’s adequacy theorem depends on the architecture being able to approximate tasks well with small supports. If a joint or monolithic baseline is much better, then the issue may still be
     representation adequacy rather than support control.
  8. Measure whether old-task gradients are actually concentrated off-support.
     The theory is about support-localized updates and low cross-Hessian coupling. We should add diagnostics closer to that:

  - gradient mass on active vs inactive columns
  - gradient mass on shared vs adaptive columns
  - per-task parameter drift by support membership

  If the goal is to get CIFAR10 continual learning actually working, the highest-probability sequence is:

  1. implement exact one-swap teacher application at task boundaries,
  2. log support changes explicitly,
  3. run teacher-only CIFAR10,
  4. then compare against the learned-controller version,
  5. then ablate shared_columns=3.

  That is the path most consistent with both the paper and the empirical evidence we now have. The current evidence says the learned signal is finally good enough; the remaining problem is that the controller
